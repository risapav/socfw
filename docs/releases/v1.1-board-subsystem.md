# v1.1 Board Subsystem Milestone

## Overview

The v1.1 board subsystem milestone delivers a complete, production-ready board modeling
layer for socfw. Boards are now first-class citizens with rich resource descriptions,
alias/profile support, connector derivation, and tight integration with validation
and RTL generation.

## Feature summary

### AC608 EP4CE15E22C8 support
Full board pack in `packs/builtin/boards/ac608_ep4ce15/board.yaml` including:
- Onboard: LEDs, buttons, UART, I2C, HDMI (TMDS), SDRAM
- Connectors: P8 (14-pin), P5 (2-pin), P6 (2-pin) GPIO headers
- Derived resources for all connectors
- Aliases, profiles, and mux groups

### Connector-derived resources
Boards define physical connectors; logical resources are derived automatically:

```yaml
resources:
  connectors:
    headers:
      P8:
        pins: [R1, T2, ...]

derived_resources:
  - name: external.headers.P8.gpio
    from: connectors.headers.P8
    role: gpio14
    top_name: HDR_P8_D
```

### Board aliases and profiles
Short alias names and named feature profiles for cleaner project YAML:

```yaml
# Board
aliases:
  leds: onboard.leds
profiles:
  minimal:
    use: [onboard.leds]

# Project
features:
  profile: minimal
# or
features:
  use: [board:@leds]
```

### Mux group enforcement
Mutually exclusive resources are validated:

```yaml
# Board
mux_groups:
  sdram_vs_headers:
    resources: [external.sdram, external.headers.P8.gpio]
    policy: mutually_exclusive
```

Error: `MUX001 resources external.sdram and external.headers.P8.gpio are mutually exclusive`

### Feature-aware pin conflict detection
PIN001 only fires when conflicting resources are both active in the project.
Sub-paths of active refs are automatically pruned to avoid false positives.

### Width adaptation RTL emission
`adapt: zero_extend`, `adapt: truncate`, `adapt: replicate` now emit actual
SystemVerilog assign statements:

```systemverilog
wire [5:0] w_blink0_ONB_LEDS;
assign PMOD_J10_LED = { 2'b0, w_blink0_ONB_LEDS };
```

### Board normalization pipeline
- Kind inference from field structure
- Legacy pin dict normalization to list format  
- Derived resource expansion before validation
- Alias warnings (BRD_ALIAS001-004)

### board.tcl emitter
External resources from active feature refs and port bindings are now
automatically emitted in `board.tcl` alongside onboard resources.

### socfw board-info command
```bash
socfw board-info ac608_ep4ce15
socfw board-info --board-file path/to/board.yaml
```

### Board selector index
The build generates `reports/board_selectors.json` for editor autocompletion support.

### New validation codes
- `BRD003` — connector-only path used as bind target  
- `BRD205` — invalid board resource direction
- `BRD_ALIAS001-004` — legacy field usage warnings
- `BRD_ALIAS404` — unknown board alias
- `MUX001` — mutually exclusive resource conflict
- `BIND006` — invalid adapt mode
- `BIND007` — adapt on inout resource

## Board resource tree helpers

New `socfw.board` package:
- `resource_tree.py` — `iter_resource_leaves`, `collect_resource_pins`, `resource_width`, `resource_direction`
- `feature_resolver.py` — resolve board: refs with alias support
- `profile_resolver.py` — expand feature profiles
- `alias_resolver.py` — resolve @alias refs
- `target_resolver.py` — unified board target resolver
- `derived_resources.py` — connector-to-resource derivation
